# 1 前言

​ 本文属于理论知识，会出现许多专有名词，了解这些名词对于网站的开发和部署都会有帮助，能提高分析、排查和解决问题的能力。计算机属于工科，比较推荐的方式是边做边学。做固然是最重要的，但学也要跟上。光学不做，那就是眼睛看，脑子会，一上手，就报废；光做不学，就难以进步，知识的深度和广度都难以提高，甚至可能都不知道为什么要这么做，不能灵活的运用知识，做起来只会事倍功半，这一点笔者本人也是深有体会。将知识编织成网络，工作会更得心应手；在实践中运用学习到的知识，对知识也会有更深的印象和理解。当然，在学习理论知识的过程中难免会遇到晦涩难懂的或者是暂时难以解释的部分，这些部分可以先不深入了解，只记住怎么去用就行。等实际上手的时候，疑惑也许就自然而然解开了。

​ 这就有点像学习乐器，你当然可以不了解乐理，只记住该按什么位置，只要练习得够多，照样可以把曲子弹出来；但学习乐理能让你对于演奏的内容有更好的理解，对于音乐的把控也会更好。

# 2 引入

​ 用户要访问网站，只需要联网，在浏览器地址栏输入网址，然后回车，网站的内容就出来了，就是这么简单。

​ 但这个过程是怎么样的呢？以及你有没有思考过以下的几个问题：

  • 不联网的时候访问一个网站会看到什么？
  • 联网后，在浏览器地址栏输入不是网址的东西再按回车会看到什么？
  • 为什么访问有些网站要很久才能显示出内容？

# 3 计算机网络

​ 先来看看最开始提的问题，当不联网时访问一个网站会看到什么，用Chrome浏览器看到的是这样的结果，他说“未连接到互联网”。

​ 那又出现了新的问题：**互联网是什么？为什么未连接到互联网就不能访问网站呢？**百度百科的说法是这样的。

​ 百度百科说，互联网是网络与网络之间以通用的协议串联成的庞大网络。做一个名词代换，计算机未连接到这个庞大的网络，就没法访问网站。**但什么是网络？**继续在百度百科搜索，发现对网络一词的解释和上网没什么关系，对它的解释是包括了很多节点和节点之间的链路。下图是网络的示意图。

​ 那就继续搜索，这次把范围缩小，搜索计算机网络，百度百科对计算机网络的定义部分很长，耐心看完不难总结出：计算机网络也是一种网络，包括了节点和链路。在计算机网络中，节点就是计算机系统和通信设备，节点之间的链路就是通信线路。而计算机网络的目的是共享资源、信息传递。

​ 在百度百科中还能看到，计算机网络可以分成局域网、城域网、广域网。分类的标准就是覆盖范围的大小，范围小的是局域网，范围稍大的是城域网，范围更大的是广域网。一个个局域网连接而成，形成了更大的局域网，甚至是城域网；城域网连接而成，构成了广域网；全世界的网络连接起来，最大的这个广域网就是互联网。

​ 至此，第一个问题就解决了，来总结梳理一下上面讲到的内容：计算机网络是由若干计算机和通信设备连接而成的计算机集合，处在同一个网络下的计算机能共享资源、传递信息。计算机网络可以根据覆盖范围，分成局域网、城域网、广域网。而互联网是最大的广域网，连接了全世界的计算机和设备。当计算机未连接到互联网，也就表示他没法和其他计算机共享资源、传递信息，而网站也是其他计算机的资源之一，所以没法访问其他人计算机中的网站。

​ 那怎么才能让自己的计算机连接到互联网，从而和其他的计算机通信呢？这个问题暂时用不着从专业的角度去思考，就想平时自己是怎么上网的就知道了：连WiFi、开流量、连网线。具体的细节之后再开一篇文章来单独说这个内容。

# 4 域名与IP地址

​ 学习了上一节可以知道，互联网是全世界计算机的集合，网站是一台计算机的资源之一，连接互联网上访问某个网站就是访问某台计算机的某个资源。而具体访问哪一台计算机的哪一份资源，我们都知道是通过网址来指定的。就像打电话一样，要想和某个人通话，就要先知道对方的电话号码。网址，可以理解成计算机的电话号码，每个网址都是唯一的。目前常用的一种写法是【xxx.xxx.xxx.xxx】，其中xxx的取值范围是0~255的之间的整数，这个就是IPv4的IP地址写法。

​ 打电话不可能记住所有联系人的电话号码，上网也一样。电话的解决办法是为联系人设置备注名，而上网则是给IP地址设置域名，例如百度的地址对应的域名就是www.baidu.com。比起一串数字，还是这种方式更能让人记住。和联系人备注有所区别的是，域名不能重复，也就是说只有一个www.baidu.com。但一个IP地址可以有多个域名，一个域名也可以对应多个IP地址。用浏览器访问www.baidu.com，其实就是去访问它对应的IP地址。

​ 那怎么知道这个域名对应的IP地址是什么呢？

​ 这就要用到DNS服务了。DNS是Domain Name System，翻译过来就是域名系统。它的功能就是记录域名和IP地址的对应关系（映射）。当浏览器访问www.baidu.com的时候，首先会在浏览器的DNS缓存（临时存放内容的区域就叫缓存）中看看有没有记录，如果有就找到了这个域名对应的IP地址就可以直接访问；如果没有，浏览器就会向计算机的DNS服务发出请求，让他帮忙找找它有没有记录；如果还是没有，就会查找本计算机的hosts文件；还是没有就会向根域名服务器发送请求。根域名服务器不知道这个域名对应的IP地址，但它知道谁知道。它就会告诉计算机DNS服务，让它去找下一个服务器来询问。以此类推，直到本地域名服务器知道了要访问的IP地址为止。如下图所示。

​ 还有一项技术叫做CDN，content delivery network。其实很简单，就是分店的思想，如果全国只有一家麦当劳，坐标北京，我广东人要想吃一趟麦当劳也太麻烦了；要是我辛辛苦苦去到北京，结果麦当劳关门了，那我心态就崩了。为了解决这个问题，麦当劳就在全国开了很多家店，并且菜单的内容都是一样的，那我就可以在广东吃到麦当劳了。如果要请求的资源所在的服务器在北京，而我人在广东，那获取资源时间就很长，甚至还会发生网络拥塞。如果厂家在全国各地都提供了服务器，那就方便各地的人访问了。CDN更适合用于存放静态内容，因为能所有服务器保持统一。CDN有负载均衡功能，自动选择最优的服务器进行响应，从而分担了单个服务器的负载。DNS可以配合CDN，把离用户最近的IP地址返回给用户。

​ 要在DNS解析时使用CDN的大致方法是：配置DNS服务，让他在解析到特定的域名时，将这个域名转交给【专门解析有CDN服务的域名的DNS服务器】。这个服务器会告诉你负载均衡服务器的地址。负载均衡服务器能干的事就是根据你的网络地址，找到离你最近并最适合的设备。配置好了之后访问过程如下图所示：

# 5 请求过程

​ 上一章讲了如何把域名解析成IP地址，但解析域名之后到浏览器显示页面之前的过程还是问号，本小节就来介绍这个过程。

​ 域名解析后，浏览器就知道了要访问的目的IP地址。但只知道IP地址是没法访问网站的，还要知道端口号。一台计算机只有一个IP，但一台计算机里面可以启动很多应用和服务，例如浏览器、QQ、微信……只知道IP没法找到具体的应用，还要靠端口号来更精准的查找。一台计算机能提供65536个端口，其中端口号在0~1023的端口，都分配给了常用的一些服务。

​ 这时候又出现一个问题：用域名访问网站的时候也没见要加端口号啊，直接就是www.baidu.com。照上面这么说，不应该还得加上端口号吗？事实上用域名访问网站的时候是有端口的，只是被省略了而已。

​ 在百度首页时，查看地址栏可以看到完整的样子：https://www.baidu.com/。前面的【https://】表示访问服务用的是https协议。协议就是规则；使用https协议，换句话说就是要按照https规则的要求来传输信息。至于https要求了什么在此就不深入讲解了。除了https以外，访问网站时常用的协议还有http。http和https都是常用的服务之一，所以他们都分配到了固定的端口，http的默认端口是80，https的默认端口为443。意思是，浏览器向服务器发送http请求时，传输的内容默认是发送到服务器的80端口的，而浏览器中输入的网址如果不加端口，默认就是向【ip地址:80端口】请求服务。

​ 当然，也有可能将项目部署到其他端口上。但为了网站访问时的便利性和安全，通常也不会想要输入端口号的。浏览器输入网址不输入端口号，默认访问80端口，那就得让访问服务器的80端口时，跳转到其他端口。解决的办法有Nginx反向代理。Nginx的内容，也会另外开一篇文章来介绍如何安装、使用、配置，在此先知道Nginx会用来干嘛就行。刚才说解决办法是Nginx反向代理，那反向代理是什么意思？代理包括正向代理和反向代理。正向代理是指客户端（也就是浏览器）去找代理。**你想和一个外国人沟通，但你们语言不通无法交流，于是你就把你想说的话告诉了翻译，由翻译来帮你向对方通信。**你找翻译帮忙传达消息就是正向代理；而反向代理就是指服务端去找人来处理请求。这个外国人会收到你的信息，也会收到其他人的信息，他会把这些信息交给不同的人帮他处理。表面上是你和外国人进行交流，事实上，是你和他找的人交流，这就是反向代理。浏览器的http请求发送到服务器的80端口，Nginx服务监听80端口，把请求分发给某台代理服务器的某个端口来处理。

​ 当请求发送到负责处理的端口之后，监听这个端口的服务就会处理请求，不同的服务有不同的请求处理方式。**网站服务会解析http请求，获取请求的参数，判断请求的具体是哪个页面，**例如b站首页是https://www.bilibili.com/，专栏页是https://www.bilibili.com/read/cv25705257/，视频页是https://www.bilibili.com/video/BV1Cu4y1U7BT/，可以看到都是https://www.bilibili.com开头，只是后面的不同。根据这些不同，服务端就会获取不同的页面来返回给浏览器。

​ 细心观察上面的视频页网址可以发现它带上了BV号。用b站的用户知道，一个BV号和一个视频一一对应，一个网址对应播放一个视频。那难道每一个视频都要写一个网页吗？从这个巨大的工作量来看很明显不是。观察可以发现，看不同的视频，虽然网址变了，但页面的结构是没有变的，变的只是内容，所以只需要写一个页面，更换页面的内容就行了。当然，更换内容也不是乱更换的，一个BV号始终对应一个视频、一个UP主、一个评论区……这些东西都是因为BV号的不同而不同。**BV号就像一个身份证（id），根据这个身份证，就能找到对应的人，知道他的年龄、性别、民族……**服务器会把每个BV号对应了哪些内容记录下来，存储到数据库中。而在数据库查找视频的配套内容，就是靠BV号这个id来作为搜索的依据。当我们进入视频页，视频页会根据BV号向服务端发送请求，查询出应该显示哪些内容，然后去获取这些资源来显示。这里有个地方要明确：浏览器请求的是只是页面的结构，页面里面的内容是由页面去请求另外的服务得到的。页面和内容是分开的，这个就是所谓的前端服务和后端服务。浏览器向前端服务发送请求，前端服务向后端服务发送请求，后端服务向数据库发送请求，数据库获取数据返回给后端服务，后端服务把数据返回给前端服务，前端服务将页面返回给浏览器。

​ 在浏览器输入网址然后回车是浏览器向前端服务发送请求，上面已经介绍过了。而前端服务向后端服务、后端服务向数据库服务发送请求，也是一个端口向另一个端口通信。前端服务向后端服务发送的请求也是http请求，不过请求的是后端服务的API接口。API接口是什么意思？再来打个比方：去医院看病首先得知道要挂哪个科室的号，这些科室是医院来划分的，不同的医院划分科室可能有所不同，病人可以看到科室的分类然后自己选择挂哪个号，每个科室的医生都有对应的知识储备来为病人看病。假如一个人喉咙痛，他看可以挂的科室里面有外科，有骨科，有耳鼻喉科……他挂了耳鼻喉科，耳鼻喉科的医生就会帮他看病，最后给病人开喉咙痛的药。API接口，大致可以理解成科室；前端请求API接口，大致可以理解成挂某个科室的号；后端返回响应，大致可以理解成开药。前端服务有啥需求就会请求哪个后端接口。接口名、接口需要的参数等信息则是写到一份接口文档中，供前端服务和后端服务的开发者一同阅读。后端服务会请求访问数据库，操作或查询数据库，然后将操作的结果返回给前端。

​ 接下来，还有最后的一步：浏览器解析请求回来的响应内容。这部分在上一篇文章《认识并编写简单网页》中也有讲过：浏览器会按照自己的方式来阅读、解析网页的内容，然后将内容呈现出来。现在就来讲一讲浏览器是如何解析和呈现的。现在我们常用的浏览器都是多进程浏览器，包括浏览器主进程、渲染进程、插件进程、GPU进程。主进程负责管理各个tab标签、管理和下载网络资源。而渲染进程负责html、css、js文件的解析、渲染、事件处理，一般来说，每打开一个标签页，就会占用一个渲染进程。浏览器收到响应后会把数据交给渲染进程处理。渲染进程开始处理：

  1. **根据html文件生成DOM结构树。**DOM是document object model，文档对象模型，将html文件表示成树形结构。**树形结构是什么呢？**生活中的树往往是一个树干分出许多树枝，树枝也可以再分，是一个多层的结构。**树形结构第一层有一个根节点，第二层就是根节点的儿子们。第三层就是第二层节点的儿子们。**在DOM结构树中，html是根节点，head和body是第二层的子节点，再往下就是第三层……

  2. 把css文件解析成CSSOM，过程和构建DOM类似。

  3. 有了上面的两棵树之后，合并生成渲染树。渲染树的节点被称为渲染对象，渲染对象是一个个矩形，和DOM元素对应，不可见的DOM元素不会被插入渲染树。

  4. 生成渲染树之后，根据渲染树来进行布局，生成一个个图层。

  5. 绘制图层。

  6. 整合所有图层，得到页面。
